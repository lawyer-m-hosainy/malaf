# دليل هيكل وقواعد بيانات منصة مَلَف (Malaf) ⚖️ 
 
هذا المستند يوثق بنية قاعدة البيانات، السياسات الأمنية، وإجراءات الصيانة لضمان استمرارية وكفاءة المنصة. 
 
## 1. الهيكل العام (Schema Design) 🏗️ 
تعتمد المنصة على قاعدة بيانات **PostgreSQL** مستضافة على **Supabase**. تم تصميم الهيكل باتباع مبادئ التطبيع (Normalization) لضمان سلامة البيانات ومنع التكرار. 
 
### الجداول الأساسية: 
- `organizations`: تمثل مكاتب المحاماة (Tenants). 
- `profiles`: الملفات الشخصية للمحامين والموظفين. 
- `cases`: سجلات القضايا والتقاضي. 
- `clients`: بيانات الموكلين والتوكيلات. 
- `sessions`: جلسات المحاكم والمواعيد المرتبطة. 
 
### الجداول المالية: 
- `invoices` & `eta_invoices`: الفواتير الداخلية والإلكترونية. 
- `trust_transactions`: سجلات الأمانات والتحصيل. 
- `payment_plans`: خطط الدفع والأقساط المجدولة. 
 
## 2. إجراءات الأمان (Security) 🔒 
### عزل البيانات (Multi-tenancy): 
يتم عزل بيانات كل مكتب بشكل صارم باستخدام **Row Level Security (RLS)**.  
- يتم التحقق من هوية المكتب عبر دالة `get_user_org_id()`. 
- لا يمكن لأي مستخدم الوصول لسجل لا يطابق `organization_id` الخاص بمكتبه. 
 
### التشفير (Encryption): 
- يتم تشفير البيانات الحساسة (مثل توكنات API الواتساب) باستخدام **AES-256-GCM**. 
- منطق التشفير يتم في الخادم عبر **Supabase Edge Functions** لضمان عدم تسريب المفاتيح للمتصفح. 
 
## 3. تحسين الأداء (Performance) 🚀 
- **الفهارس (Indexes):** تم إنشاء فهارس على كافة حقول الربط (`org_id`, `case_id`, `client_id`) وحقول البحث المتكرر (الاسم، الهاتف، الرقم القومي). 
- **الاستعلامات:** تم إضافة عمود `org_id` مباشرة للجداول كثيرة الاستخدام (مثل `sessions`) لتجنب عمليات `JOIN` المكلفة. 
- **التدقيق التلقائي:** يتم تسجيل كافة العمليات الحساسة آلياً عبر `Database Triggers` في جدول `audit_logs`. 
 
## 4. النسخ الاحتياطي والاستعادة (Backup & Recovery) 💾 
- **النسخ الآلي:** يقوم Supabase بأخذ نسخ احتياطية يومية تلقائياً. 
- **نقطة الاستعادة:** يمكن استعادة قاعدة البيانات لأي نقطة زمنية (Point-in-Time Recovery) خلال آخر 7 أيام (حسب خطة الاستضافة). 
- **الاختبار:** يُنصح بإجراء اختبار استعادة دوري على بيئة "Staging" كل 3 أشهر. 
 
## 5. قيود السلامة (Integrity Constraints) ✅ 
- **الحذف الناعم (Soft Delete):** السجلات القانونية والمالية لا تُحذف نهائياً بل يتم وسمها بـ `deleted_at`. 
- **التحقق من البيانات:** توجد قيود `CHECK` تضمن أن المبالغ المالية دائماً موجبة وأن التواريخ منطقية. 
- **فرض الحصص:** مشغلات (Triggers) تمنع إضافة سجلات تتجاوز حدود باقة الاشتراك النشطة للمكتب. 
 
## 6. إجراءات الصيانة (Maintenance) 🛠️ 
1. **تحديث الـ Schema:** يجب أن يتم دائماً عبر ملفات `sql` في مجلد `supabase/migrations`. 
2. **مراقبة الأداء:** استخدام `Supabase Dashboard → Database → Reports` لمراقبة الاستعلامات البطيئة. 
3. **مراجعة السجلات:** مراجعة جدول `audit_logs` دورياً لاكتشاف أي نشاط مريب.

*تم إعداد هذا الهيكل ليكون قابلاً للتوسع (Scalable) ليتحمل آلاف المكاتب وملايين السجلات.*
