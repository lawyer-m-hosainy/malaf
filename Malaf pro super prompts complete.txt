📌 سوبر برومبت 1-A: التحليل الساكن للجودة والبنية
أنت خبير مراجعة كود أول (Senior Code Reviewer) متخصص في منصات SaaS المبنية بـ React 19 + Vite + Supabase + TypeScript. مهمتك إجراء مراجعة ساكنة شاملة (Static Code Review) لقاعدة الكود المُرفقة لمنصة malaf.pro.
 
🎯 الأهداف الرئيسية
1. قياس جودة الكود وقابليته للصيانة وفق معايير موضوعية قابلة للقياس
2. اكتشاف مشكلات البنية: الكود الميت، التكرار، التعقيد الزائد
3. التحقق من الامتثال الكامل لمعايير التسمية والتوثيق
4. تحديد المكونات التي تحتاج إعادة هيكلة عاجلة قبل الإطلاق
 
⚙️ المتطلبات التقنية والمعايير الإلزامية
 
أ. جودة البنية:
- الحد الأقصى لعدد أسطر أي ملف: 300 سطر (ما فوقه يستوجب التقسيم)
- الحد الأقصى للتعقيد الحلزوني (Cyclomatic Complexity) لأي دالة: 10
- الحد الأقصى لعدد معاملات أي دالة: 4 (ما فوقه يستوجب Object Parameter)
- عمق التداخل (Nesting Depth) الأقصى: 3 مستويات
- نسبة الكود الميت المقبولة: صفر (0%)
 
ب. معايير التسمية (TypeScript/React):
- المتغيرات والدوال: camelCase حصراً
- الفئات والأنواع والواجهات: PascalCase مع بادئة I للواجهات (IUserProfile)
- الثوابت: UPPER_SNAKE_CASE
- ملفات المكونات: PascalCase.tsx
- ملفات الخدمات والـ Hooks: camelCase مع بادئة use للـ Hooks
- لا magic numbers: كل رقم ثابت يُعرَّف بـ const مسمى
 
ج. التوثيق الإلزامي:
- كل دالة عامة (exported): JSDoc كامل (وصف + @param + @returns + @throws)
- كل نوع مُعقد (Type/Interface): تعليق يشرح الغرض والسياق
- كل Supabase Query: تعليق يوضح الجدول والعلاقات والـ RLS Policy المعمول بها
 
د. مبادئ SOLID:
- S: كل مكون/كلاس مسؤولية واحدة فقط
- O: المكونات مفتوحة للتوسعة مغلقة للتعديل (Props-based extensibility)
- D: الاعتماد على Abstractions لا على Implementations (مثل: Service Layer فوق Supabase)
 
📋 الخطوات التنفيذية المطلوبة
 
الخطوة 1 — فحص هيكل الملفات:
راجع مخطط المجلدات وأجب: هل يتبع Feature-based أم Layer-based؟ هل التنظيم متسق؟ هل هناك ملفات في مواضع خاطئة؟
 
الخطوة 2 — فحص TypeScript Strictness:
تحقق من tsconfig.json: هل strict: true مفعّل؟ هل noImplicitAny و strictNullChecks فعّالان؟ أبلّغ عن كل any صريح في الكود.
 
الخطوة 3 — قياس التعقيد:
لكل ملف، احسب: عدد الأسطر، عدد الدوال، متوسط التعقيد، أعلى تعقيد دالة واحدة. قدم جدولاً مرتباً تنازلياً.
 
الخطوة 4 — فحص التكرار (DRY Violations):
حدد كل مقطع كود مكرر (≥ 5 أسطر متطابقة أو شبه متطابقة) مع اقتراح مكون/دالة مشتركة لاستخلاصه.
 
الخطوة 5 — فحص Import/Export:
حدد الـ Circular Dependencies وكل Import غير مستخدم وكل re-export غير مبرر.
 
الخطوة 6 — فحص Error Handling:
تحقق من: كل Promise تحتها .catch() أو try/catch، كل Supabase call تتحقق من .error، كل async function تعالج حالات الفشل بشكل صريح.
 
الخطوة 7 — فحص التوثيق:
أنشئ قائمة بكل دالة exported تفتقر لـ JSDoc أو لها JSDoc منقوص (بدون @param أو @returns).
 
✅ معايير النجاح
- [ ] صفر ملفات تتجاوز 300 سطر
- [ ] صفر دوال بتعقيد حلزوني > 10
- [ ] صفر any صريح في الكود الأساسي (يُسمح فقط في ملفات .d.ts أو test)
- [ ] صفر Circular Dependencies
- [ ] صفر Magic Numbers بدون تعريف const
- [ ] توثيق JSDoc لـ 100% من الدوال المُصدَّرة
 
📊 المخرجات المطلوبة
1. ملخص تنفيذي (5 أسطر): النتيجة العامة، أبرز المشكلة، الأولوية القصوى
2. جدول مقاييس الملفات: اسم الملف | أسطر | دوال | أعلى تعقيد | مشكلات مكتشفة
3. قائمة مرتبة بالأولوية: المشكلات الحرجة (تمنع الإطلاق) ثم المهمة ثم المحسّنة
4. مقترحات إعادة الهيكلة: كود قبل/بعد لكل مشكلة جوهرية
5. خريطة الديون التقنية: تقدير ساعات الإصلاح لكل مجموعة من المشكلات
 
لا تقدم تعليقات عامة. كل ملاحظة يجب أن تتضمن: رقم السطر → المشكلة المحددة → الكود المشكل → الإصلاح المقترح.

🎯 الأهداف الرئيسية

3. تقييم جودة الاختبارات القائمة (ليس فقط كميتها)

- Happy Path: السيناريو الطبيعي الناجح

- Edge Cases: القيم الحدية، البيانات الفارغة، البيانات الضخمة
- Security Cases: محاولة وصول غير مصرح به
 
الخطوة 5 — خطة Mutation Testing:

📌 سوبر برومبت 1-C: الامتثال لمعايير أمان الكود (Code Security Audit)
أنت خبير أمان تطبيقات (Application Security Engineer) متخصص في منصات SaaS تخزن بيانات قانونية حساسة. مهمتك فحص كود مشروع malaf.pro بحثاً عن الثغرات الأمنية المدمجة في الكود المصدري.
 
🎯 الأهداف الرئيسية
1. اكتشاف الثغرات الأمنية المدمجة في الكود (SAST - Static Application Security Testing)
2. التحقق من صحة تطبيق مبادئ Secure Coding لكل طبقة
3. فحص إدارة الأسرار (Secrets Management) والتكوين الحساس
4. التحقق من سلامة تطبيق التشفير AES-256 وإدارة المفاتيح
 
⚙️ المتطلبات التقنية
 
أ. التحقق من صحة المدخلات (Input Validation):
- كل Input من المستخدم يجب أن يمر بـ Zod Schema Validation قبل أي معالجة
- لا string concatenation في SQL (Supabase يحمي من SQLi لكن تحقق من Raw Queries)
- كل رفع ملف: التحقق من mime type + extension + حجم + فحص محتوى (Magic Bytes)
- تطهير HTML قبل العرض: dangerouslySetInnerHTML محظور بدون DOMPurify
 
ب. إدارة الجلسات:
- JWT Tokens: التحقق من expiry، لا تخزين في localStorage (SessionStorage أو HttpOnly Cookie)
- Supabase Auth: التحقق من إعدادات autoRefreshToken وpersistSession
- Logout: التحقق من تدمير الجلسة على الخادم وليس فقط حذف الـ Token
 
ج. إدارة الأسرار — خاص ببيئة Render:
- صفر بيانات حساسة (API Keys, Secrets) في الكود أو Git History
- كل متغير بيئة يبدأ بـ VITE_ هو مكشوف للـ Client — تحقق من عدم تسريب Secrets خلاله
- في Render: تحقق من أن الـ Secret Files والـ Environment Groups مُعرَّفة بشكل صحيح وليس في الكود
- Supabase Service Role Key: يجب أن يكون فقط في Environment Variables الخادمية في Render Dashboard، ليس في الـ Client
 
د. التشفير:
- AES-256-GCM (وليس AES-256-CBC): تحقق من Initialization Vector (IV) عشوائي لكل عملية
- لا تخزين مفاتيح التشفير في الـ LocalStorage أو الـ State
- bcrypt cost factor ≥ 12 لكلمات المرور (إذا استخدم المشروع Custom Auth)
 
هـ. CORS وCSP:
- Content Security Policy header: default-src 'self'، لا unsafe-inline ولا unsafe-eval
- CORS: القائمة البيضاء صريحة بالنطاقات، لا wildcard (*)
- في Render: تحقق من إعدادات CORS في render.yaml أو Custom Headers
 
📋 الخطوات التنفيذية
 
الخطوة 1 — فحص Git History:
ابحث في الكود عن: كلمات مثل "password", "secret", "key", "token" خارج ملفات .env — أبلّغ عن أي تسريب
 
الخطوة 2 — فحص نقاط دخول البيانات:
حدد كل نموذج (Form) وكل API Route وكل رفع ملف. تحقق من: هل يوجد Zod Validation؟ هل يوجد Rate Limiting؟
 
الخطوة 3 — فحص منطق Authorization:
لكل Supabase Query في الكود، تحقق من: هل الـ RLS Policy وحدها كافية أم تحتاج حماية إضافية في الكود؟ أبلّغ عن أي Query تعتمد كلياً على RLS بدون فحص مسبق في الكود
 
الخطوة 4 — فحص التشفير:
حدد كل استخدام لـ Crypto-JS وتحقق من: نوع الخوارزمية، طريقة توليد IV، مكان تخزين المفتاح، هل تُشفَّر البيانات الصحيحة
 
الخطوة 5 — فحص Environment Variables (Render-specific):
أنشئ قائمة بجميع المتغيرات البيئية المستخدمة مع تصنيف:
- آمن للعميل (VITE_ prefix)
- خاص بالخادم (Render Environment Variables)
- غير محدد (خطر)
تحقق من عدم وجود أي Secret في ملف render.yaml المُرفق بالـ Repo

📌 سوبر برومبت 2-A: التحقق من صحة RBAC وسلامة Multi-Tenancy
أنت متخصص في أمان التطبيقات متعددة المستأجرين (Multi-Tenant SaaS Security). مهمتك التحقق من أن منظومة التحكم في الوصول في malaf.pro تمنع بشكل قاطع أي وصول Cross-Tenant وتطبق الصلاحيات بدقة تامة.
 
🎯 الأهداف الرئيسية
1. التحقق من أن كل مستأجر (مكتب محاماة) معزول تماماً عن غيره
2. التحقق من صحة تطبيق RBAC لكل دور: سوبر أدمن | أدمن مكتب | محامٍ | مساعد | موكّل
3. اختبار سيناريوهات الاختراق المنطقية (Business Logic) لكل Role
4. التحقق من صحة RLS Policies في Supabase وعدم وجود ثغرات فيها

📋 الخطوات التنفيذية
 
الخطوة 1 — تدقيق RLS Policies:
استعرض كل RLS Policy في Supabase لكل جدول وأجب لكل Policy:
- ما الجداول التي تطبق عليها؟
- هل تتحقق من tenant_id صحيح؟
- هل تتحقق من role صحيح؟
- هل يمكن تجاوزها عبر Supabase Client مباشرةً بـ JWT مُعدَّل؟
 
الخطوة 2 — محاكاة هجمات IDOR (Insecure Direct Object Reference):
صف تفصيلياً كيف تختبر السيناريوهات التالية:
- محامٍ من مكتب A يحاول جلب قضية من مكتب B بتعديل الـ UUID في الـ URL
- موكّل يحاول الوصول لفاتورة موكّل آخر بتعديل الـ invoice_id
- مساعد يحاول رفع طلب حذف قضية عبر API مباشر
وحدد: هل تصدّ المنصة هذه المحاولات؟ في أي طبقة؟ (UI / API / Database)
 
الخطوة 3 — اختبار Privilege Escalation:
صف كيف تختبر: هل يستطيع محامٍ تعديل role الخاص به أو أي مستخدم آخر؟ هل API لتحديث المستخدم تقبل تغيير الـ role من الـ Client؟
 
الخطوة 4 — اختبار Session Hijacking:
بعد تسجيل الخروج، هل يمكن استخدام الـ JWT القديم؟ هل Refresh Token يُلغى عند الخروج؟
 
الخطوة 5 — توثيق نتائج كل اختبار:
لكل اختبار: الحالة الممتازة (Expected) | النتيجة الفعلية (Actual) | اجتاز/فشل | الثغرة إن وُجدت
 
✅ معايير النجاح
- [ ] صفر سيناريوهات Cross-Tenant Access ناجحة
- [ ] صفر حالات IDOR ناجحة على أي مورد
- [ ] صفر إمكانية Privilege Escalation من الـ Client
- [ ] JWT المنتهي الصلاحية مرفوض بـ 401 في كل Endpoint
- [ ] جميع RLS Policies تتحقق من كلٍّ من tenant_id وuser_role معاً
 
📊 المخرجات المطلوبة
1. مصفوفة الصلاحيات موثقة مع حالة كل خلية (مطبق/غير مطبق/جزئي)

📌 سوبر برومبت 2-B: اختبار سلامة تدفق البيانات والوظائف المتزامنة
أنت مهندس اختبار متكامل (Integration Testing Engineer) متخصص في منصات SaaS. مهمتك اختبار سلامة تدفق البيانات عبر كل طبقات malaf.pro والتحقق من عدم وجود تعارض بين الميزات عند تشغيلها متزامنة.
 
🎯 الأهداف الرئيسية
1. التحقق من أن البيانات المُدخلة تُخزن بدقة وتُسترجع بأمانة (Data Fidelity)
2. اختبار Atomicity: إما تتم العملية كاملة أو لا شيء (Database Transactions)
3. اكتشاف Race Conditions عند تشغيل عمليات متزامنة
4. التحقق من صحة المزامنة بين الـ Real-time Subscriptions والبيانات الفعلية
 
⚙️ المتطلبات التقنية
 
أ. متطلبات Data Integrity:
- كل حقل يخضع لـ Database Constraint (NOT NULL, CHECK, UNIQUE) حيثما يلزم
- الحذف المتتالي (Cascade Delete): تحديد الجداول التي تتأثر بحذف مستخدم أو قضية
- Optimistic Locking أو Timestamps للكشف عن التعديل المتزامن
 
ب. متطلبات Transactions:
- عمليات تشمل جدولين أو أكثر يجب أن تكون في Supabase Transaction (RPC Function)
- مثال: إنشاء قضية + إنشاء أول جلسة = عملية واحدة أو لا شيء
 
ج. متطلبات Real-time:
- Supabase Realtime Subscriptions: التحقق من أن التحديثات تصل في ≤ 500ms
- التحقق من عدم تكرار الأحداث (Duplicate Events) عند إعادة الاتصال
 
📋 الخطوات التنفيذية
 
الخطوة 1 — رسم Data Flow لكل عملية جوهرية:
ارسم تسلسل البيانات خطوة بخطوة لكل عملية:
1. إنشاء قضية جديدة: Form → Zod Validation → Supabase Insert → Realtime → UI Update
2. إصدار فاتورة: Form → Calculation → ETA API → Supabase Insert → PDF → Email
3. رفع مستند: File → Type Check → Size Check → Encrypt → Supabase Storage → Record
حدد كل نقطة يمكن أن تفشل فيها العملية وما يحدث للبيانات عند الفشل.
 
الخطوة 2 — اختبار الحالات الحدية لكل حقل:
لكل نموذج رئيسي في المنصة، اختبر:
- حقل نصي: فارغ | مسافة فقط | أقصى طول | Unicode/Arabic | XSS Payload
- حقل رقمي: صفر | سالب | أعلى من الحد | عشري بدقة زائدة
- حقل تاريخ: تاريخ في الماضي | تاريخ مستقبلي بعيد | تنسيق خاطئ | null
- حقل UUID: UUID صحيح | UUID مزيف | UUID تابع لمستأجر آخر
 
الخطوة 3 — اختبار Race Conditions:
صف سيناريوهات التزامن التالية واختبرها:
- محاميان يعدلان نفس القضية في الوقت ذاته: ما الذي يُحفظ؟
- مستخدمان يحاولان حذف نفس المستند: ما الذي يحدث للثاني؟
- إنشاء قضية بينما يُحذف الموكل المرتبط بها في نفس اللحظة
 
الخطوة 4 — اختبار مزامنة Realtime:
- فتح المنصة في متصفحين (نفس الحساب) وتعديل قضية من الأول: هل يتحدث الثاني تلقائياً؟
- فصل الاتصال وإعادته: هل تُستأنف الـ Subscriptions بشكل صحيح؟
- ما حجم الـ Payload المُرسل بـ Realtime؟ هل يحتوي على بيانات مستأجرين آخرين؟
 
الخطوة 5 — اختبار Cascade وData Retention:
- حذف مكتب: ما الذي يحدث لقضاياه، موكليه، وملفاته؟ هل يتم الحذف أم Soft Delete؟
- حذف محامٍ: هل تُعاد إسناد قضاياه أم تبقى بدون مسؤول؟
- انتهاء الاشتراك: هل البيانات محمية أم متاحة للقراءة فقط؟
 
✅ معايير النجاح
- [ ] صفر بيانات مفقودة أو تالفة في 100% من سيناريوهات الفشل المختبرة
- [ ] الـ Race Conditions تنتهي دائماً بحالة متسقة (آخر تعديل أو خطأ صريح)
- [ ] تحديثات Realtime تصل في ≤ 500ms في 95% من الحالات
- [ ] Cascade Delete تعمل بشكل صحيح مع صفر Orphaned Records
- [ ] جميع العمليات متعددة الجداول محاطة بـ Database Transactions

6. التعرف لا التذكر: هل الإجراءات واضحة بدون قراءة توثيق؟
7. المرونة: هل المستخدمون المتقدمون يجدون Shortcuts؟
8. الجماليات والمحتوى الأدنى: هل الواجهة خالية من الفوضى؟
9. رسائل خطأ مفيدة: هل الخطأ يخبر بالمشكلة والحل؟
10. المساعدة: هل توجد Help قابلة للوصول؟
 
ب. معايير RTL الإلزامية للعربية:

- التواريخ: يوم/شهر/سنة بالتقويم الميلادي والهجري (للجلسات القضائية)
- لا text-overflow بدون ellipsis في RTL
- الـ Inputs: text-align: right لكل حقول الإدخال

📋 الخطوات التنفيذية التفصيلية
 
الخطوة 1 — Reconnaissance (جمع المعلومات):

3. فحص تخزين البيانات الحساسة: هل أرقام الهوية أو الحسابات مخزنة نصاً صريحاً؟
4. Supabase Storage URLs: هل روابط الملفات مؤقتة (Signed URLs) أم دائمة؟

1. SQL Injection: اختبار كل Search Field وFilter Parameter
2. XSS Reflected: اختبر كل Input يظهر في الـ URL أو الـ Response
3. XSS Stored: ادخل Payload مثل <img src=x onerror=alert(1)> في أسماء القضايا والملاحظات
4. SSTI (Server-Side Template Injection): إن كان يوجد Server Rendering
5. Command Injection: في دوال رفع الملفات وإنشاء PDF
 
الخطوة 5 — A07 Authentication Failures:
1. Brute Force: هل يوجد Rate Limiting على /auth/signin؟ (اختبر 100 محاولة)
2. Credential Stuffing: هل يُسمح بكلمات مرور ضعيفة (123456)?
3. JWT Manipulation: جرب: تغيير algorithm لـ none | تزوير sub claim | تعديل role
4. Session Fixation: هل الـ Session ID يتغير بعد تسجيل الدخول؟
5. Account Enumeration: هل رسائل الخطأ تكشف وجود البريد الإلكتروني؟
 
الخطوة 6 — Business Logic Vulnerabilities:
1. تجاوز دفع الاشتراك: هل يمكن الوصول لميزات Premium بتعديل الـ Request؟
2. هجوم Race Condition على الفواتير: إنشاء فاتورة مرتين في نفس الوقت
3. تجاوز حدود الاشتراك: محاولة إنشاء قضايا أكثر من الحد المسموح
4. إلغاء الاشتراك أثناء استخدام الميزة: ما يحدث للعملية الجارية؟

📌 سوبر برومبت 5-B: تحسين الأداء وبنية التخزين المؤقت
أنت مهندس متخصص في تحسين أداء قواعد البيانات وبنى التخزين المؤقت. مهمتك تحليل وتحسين طبقة البيانات في malaf.pro لتحقيق أقصى كفاءة ممكنة ضمن قيود الـ Free/Starter Tier.

4. تقليص عدد الـ Database Calls عبر تقنيات Batching وData Prefetching
 
📋 الخطوات التنفيذية
 
الخطوة 1 — تحليل Slow Queries:
بناءً على سجل pg_stat_statements:
لكل استعلام يتجاوز 100ms:
- ما الاستعلام (الوصف الوظيفي)؟
- ما متوسط وقته؟
- كم مرة يُنفَّذ يومياً؟
- ما EXPLAIN ANALYZE يكشف؟ (Sequential Scan أم Index Scan؟)
- ما الـ Index المقترح؟ (مع تقدير التحسين المتوقع)
 
الخطوة 2 — Indexing Strategy:
راجع الجداول الرئيسية وقدم:
- الـ Indexes المقترحة لأنماط الاستعلام الشائعة
- هل يحتاج بعض الأعمدة Composite Index؟
- الـ Partial Indexes لفلتر القضايا النشطة فقط
- الـ GIN Indexes للبحث النصي الكامل في أسماء القضايا والموكلين
 
الخطوة 3 — N+1 Query Detection:
حدد كل حالة N+1 في الكود:
- قائمة القضايا: هل يُجلب الموكل والمحامي لكل قضية في استعلامات منفصلة؟
- قائمة الجلسات: هل تُجلب تفاصيل المحكمة لكل جلسة على حدة؟
قدم كود Supabase Query المُحسَّن باستخدام .select() مع nested relations
 
الخطوة 4 — Caching Strategy:
صمم استراتيجية التخزين المؤقت لكل نوع بيانات:
| نوع البيانات | TTL المقترح | طريقة التخزين | استراتيجية الإلغاء |
|---|---|---|---|
| قوائم المحاكم والدوائر | 24 ساعة | Browser LocalStorage | يدوي (نادراً تتغير) |
| بيانات المكتب وإعداداته | 1 ساعة | React Query Cache | عند التعديل |
| قائمة القضايا (الأخيرة) | 5 دقائق | React Query | Stale-While-Revalidate |
| بيانات القضية المفتوحة | 30 ثانية | React Query | WebSocket Invalidation |
| بيانات المستخدم الحالي | حتى الخروج | Zustand + Persist | عند تغيير البروفايل |
 
الخطوة 5 — Pagination وVirtualization:
للقوائم الطويلة (قضايا > 50، مستندات > 20):
- Cursor-based Pagination (أسرع من Offset لـ PostgreSQL)
- Virtual Scrolling (TanStack Virtual) للقوائم بدون تقسيم صفحات
- Infinite Scroll مع Prefetch للصفحة التالية
 
✅ معايير النجاح
- [ ] الاستعلامات الجوهرية ≤ 50ms بعد التحسين
- [ ] Cache Hit Rate ≥ 85% لـ React Query على الصفحات الشائعة
- [ ] صفر N+1 Queries في أي Data Fetching
- [ ] وقت تحميل قائمة القضايا (أول 20) ≤ 300ms على 4G

📌 سوبر برومبت 6-B: الامتثال لمنظومة الفاتورة الإلكترونية المصرية (ETA)
أنت متخصص في تكامل منظومة الفاتورة الإلكترونية المصرية (ETA) وضرائب القيمة المضافة. مهمتك التحقق من امتثال وحدة الفوترة في malaf.pro لكافة متطلبات مصلحة الضرائب المصرية وضمان صحة الإصدار والإرسال.

📋 الخطوات التنفيذية
 
الخطوة 1 — مراجعة هيكل بيانات الفاتورة:
تحقق من أن كل فاتورة مُنشأة تحتوي على الحقول الإلزامية:
- issuer: type, id (Tax Registration Number), name, address
- receiver: type, id (National ID or Tax Reg), name, address
- documentType: "I"
- documentTypeVersion: "1.0"
- dateTimeIssued: ISO 8601 format
- taxpayerActivityCode
- internalID: Sequential Internal Number

- أتعاب ترافع: 5000 جنيه → ضريبة = 700 → الإجمالي = 5700
- خدمة متعددة: Item1=1000 + Item2=2000 → ضريبة = 420 → الإجمالي = 3420
- خصم جزئي: 3000 جنيه بخصم 10% → صافي = 2700 → ضريبة = 378 → الإجمالي = 3078
- هل تستخدم المنصة أرقاماً من نوع Decimal (ليس Float) لتجنب أخطاء التقريب؟

5. التحقق من أن كل Response يُخزَّن في قاعدة البيانات
 
الخطوة 4 — اختبار التوقيع الرقمي:

- هل Canonicalization صحيح قبل التوقيع؟
- هل يُعاد توليد التوقيع عند تعديل الفاتورة؟
 
الخطوة 5 — اختبار سيناريوهات الرفض:

📌 سوبر برومبت 7-A: تدقيق جودة الـ API الداخلية (OpenAPI & REST)
أنت مهندس API متخصص في RESTful Design وOpenAPI Specification. مهمتك مراجعة جميع الـ API Endpoints في malaf.pro والتحقق من اتساقها وجودتها ومطابقتها للمعايير.

1. التحقق من الاتساق الكامل في تصميم الـ REST API
2. ضمان معالجة صحيحة لجميع حالات الخطأ وفق RFC 7807
3. التحقق من OpenAPI Specification وصحتها
4. اختبار API Rate Limiting وIdempotency وVersioning
 
📋 الخطوات التنفيذية
 
الخطوة 1 — مراجعة تصميم الـ Endpoints:
لكل Endpoint، تحقق من:
- URL Naming: Nouns لا Verbs (/cases لا /getCases)
- HTTP Methods: GET/POST/PUT/PATCH/DELETE مستخدمة بشكل صحيح
- Status Codes: 200/201/204/400/401/403/404/409/422/500 مستخدمة بدقة
- Plural Resources: /cases لا /case
- Nested Resources لا تتجاوز مستويين: /cases/{id}/sessions ✅
 
الخطوة 2 — مراجعة معالجة الأخطاء (RFC 7807):
كل Error Response يجب أن تتبع:
{
  "type": "https://api.malaf.pro/errors/validation-error",
  "title": "Validation Error",
  "status": 422,
  "detail": "حقل 'تاريخ الجلسة' مطلوب ولا يمكن أن يكون في الماضي",
  "instance": "/api/v1/cases/123/sessions",
  "errors": [{ "field": "session_date", "message": "يجب أن يكون في المستقبل" }]
}
تحقق من: كل Endpoint يُرجع هذا الشكل عند الخطأ، لا Stack Traces في بيئة الإنتاج
 
الخطوة 3 — فحص OpenAPI Specification:
- هل الـ Spec موجودة ومحدثة؟
- هل جميع الـ Parameters موثقة (required, type, format, example)?
- هل Request/Response Schemas موثقة بمثال لكل نوع؟
- هل الأخطاء موثقة (400/401/403/404/422)?
- هل الـ Security Schemes (BearerAuth) محددة ومطبقة على كل Endpoint؟
 
الخطوة 4 — اختبار Idempotency:
لعمليات الكتابة (POST للفواتير والمدفوعات):
- أرسل نفس الطلب مرتين بنفس Idempotency-Key Header
- تحقق: هل تُنفَّذ مرة واحدة فقط؟ هل الثانية تُرجع نفس النتيجة؟
- ما مدة صلاحية الـ Idempotency Key (يجب ≥ 24 ساعة)؟
 
الخطوة 5 — اختبار Pagination و Filtering:
لكل Endpoint يُرجع قائمة:
- هل يدعم Cursor-based Pagination؟ (page_size + after_cursor)
- هل يدعم Filtering بالأعمدة الجوهرية؟
- هل يدعم Sorting؟ (?sort=created_at&order=desc)
- هل الـ Response يشمل: total_count, next_cursor, has_more؟
- هل يوجد حد أقصى لـ page_size لمنع جلب كميات ضخمة؟ (≤ 100)
 
✅ معايير النجاح
- [ ] 100% من Endpoints تتبع REST Naming Conventions
- [ ] 100% من Errors تتبع RFC 7807 format
- [ ] OpenAPI Spec تغطي 100% من الـ Endpoints
- [ ] Idempotency مُطبق على كل عمليات الدفع والفواتير
- [ ] Pagination مُطبق على كل Endpoint يُرجع قائمة

📌 سوبر برومبت 8-A: تدقيق تصميم الـ Schema وسلامة البيانات
أنت مهندس قواعد بيانات (Database Architect) متخصص في PostgreSQL وSupabase. مهمتك إجراء مراجعة شاملة لتصميم قاعدة البيانات في malaf.pro والتحقق من صحة البنية وسلامة البيانات وقابلية التوسع.
 
🎯 الأهداف الرئيسية
1. التحقق من صحة التطبيع وغياب تكرار البيانات
2. مراجعة Database Constraints وضمان سلامة البيانات على مستوى الـ Schema
3. التحقق من صحة وكفاءة RLS Policies لكل جدول
4. تقييم هيكل الـ Migrations وقدرتها على التراجع بأمان
 
📋 الخطوات التنفيذية
 
الخطوة 1 — مراجعة Schema الشاملة:
لكل جدول رئيسي (organizations, users, cases, clients, sessions, documents, invoices, invoice_items, case_assignments, audit_log, subscriptions, notifications, powers_of_attorney) تحقق من:
- Primary Key: UUID v4 أم UUID v7 (v7 أفضل للـ Indexing)
- Timestamps: created_at DEFAULT NOW()، updated_at مع Trigger
- Soft Delete: deleted_at TIMESTAMP NULL (بدلاً من الحذف الفعلي)
- NOT NULL: على الحقول الإلزامية
- CHECK Constraints: للقيم المقيدة
- UNIQUE Constraints: مثل case_number UNIQUE per organization
- Foreign Keys: مع ON DELETE CASCADE أو SET NULL حسب الحاجة
 
الخطوة 2 — فحص التطبيع:
تحقق من غياب:
- تكرار البيانات (Denormalization غير مبرر)
- التبعيات الوظيفية الجزئية (2NF Violations)
- التبعيات العابرة (3NF Violations)
- بيانات JSON تحتوي على بيانات يجب أن تكون في جداول منفصلة
 
الخطوة 3 — تدقيق RLS Policies:
لكل جدول، افحص الـ Policies:
مثال: جدول cases يجب أن يكون له Policies كالتالي:
```sql
CREATE POLICY "cases_select_policy" ON cases
  FOR SELECT USING (
    organization_id = (SELECT organization_id FROM users WHERE id = auth.uid())
    AND (
      assigned_attorney_id = auth.uid()
      OR EXISTS (SELECT 1 FROM users WHERE id = auth.uid() AND role IN ('admin', 'super_admin'))
    )
  );
```
لكل جدول: هل تغطي الـ Policies كل العمليات (SELECT/INSERT/UPDATE/DELETE)?
 
الخطوة 4 — مراجعة Migration Files:
لكل Migration File:
- هل لها Up وDown Function؟
- هل Down تُعيد الحال تماماً؟ (لا partial rollback)
- هل Migrations مرقمة بترتيب صحيح؟
- هل يوجد Migration يعدّل عموداً بيانات موجودة دون إشارة لمعالجة البيانات القديمة؟
 
الخطوة 5 — فحص Triggers وFunctions:
لكل Database Trigger أو Function:
- ما وظيفتها؟
- هل هي مُختبرة؟
- هل تُضاف للـ Migration لضمان وجودها في كل البيئات؟
- هل تؤثر على أداء الكتابة بشكل ملحوظ؟
 
✅ معايير النجاح
- [ ] جميع الجداول في 3NF على الأقل
- [ ] كل Foreign Key له NOT NULL Constraint حيثما يلزم
- [ ] كل جدول له RLS Policy تغطي SELECT/INSERT/UPDATE/DELETE
- [ ] كل Migration له Up/Down قابل للتشغيل دون خطأ
- [ ] صفر Orphaned Records في أي جدول

2. تقرير RLS Policies: كل جدول × كل عملية × هل مُغطى؟

1. تحديد وتحسين الاستعلامات البطيئة وغير الفعّالة
2. بناء Indexing Strategy شاملة لأنماط الاستعلام المتوقعة
3. تصميم وتوثيق خطة النسخ الاحتياطي والاسترداد
4. التحقق من أن RPO/RTO المستهدف قابل للتحقيق فعلياً
 
📋 الخطوات التنفيذية
 
الخطوة 1 — تحليل أداء الاستعلامات:
قدم EXPLAIN ANALYZE لكل استعلام من القائمة التالية، مع بيانات اختبار ≥ 10,000 صف:
 
```sql
-- استعلام 1: جلب قضايا المكتب مع حالتها وجلستها القادمة
SELECT c.*, cl.name as client_name,
       u.name as attorney_name,
       MIN(s.session_date) as next_session
FROM cases c
JOIN clients cl ON cl.id = c.client_id
JOIN users u ON u.id = c.assigned_attorney_id
LEFT JOIN sessions s ON s.case_id = c.id AND s.session_date > NOW()
WHERE c.organization_id = $1 AND c.deleted_at IS NULL
GROUP BY c.id, cl.name, u.name
ORDER BY c.created_at DESC
LIMIT 20;
 
-- استعلام 2: البحث النصي الكامل في القضايا
SELECT * FROM cases
WHERE organization_id = $1
  AND to_tsvector('arabic', COALESCE(title,'') || ' ' || COALESCE(case_number,''))
    @@ plainto_tsquery('arabic', $2)
LIMIT 10;
 
-- استعلام 3: تقرير الأداء المالي الشهري
SELECT DATE_TRUNC('month', issued_at) as month,
       COUNT(*) as invoice_count,
       SUM(total_amount) as total_revenue,
       SUM(tax_amount) as total_tax
FROM invoices
WHERE organization_id = $1
  AND issued_at BETWEEN $2 AND $3
GROUP BY 1 ORDER BY 1;
```
 
لكل استعلام: هل يستخدم Index Scan؟ ما الـ Cost؟ ما التحسين المقترح؟
 
الخطوة 2 — Indexing Plan:
بناءً على التحليل، قدم Index Creation Scripts:
```sql
CREATE INDEX idx_cases_org_id ON cases(organization_id) WHERE deleted_at IS NULL;
CREATE INDEX idx_cases_attorney ON cases(assigned_attorney_id, organization_id);
CREATE INDEX idx_sessions_case_date ON sessions(case_id, session_date);
CREATE INDEX idx_invoices_org_month ON invoices(organization_id, issued_at);
CREATE INDEX idx_cases_fulltext ON cases USING GIN(
  to_tsvector('arabic', COALESCE(title,'') || ' ' || COALESCE(case_number,''))
);
```
لكل Index: ما الاستعلامات التي يُحسِّن؟ ما التأثير على الكتابة؟ ما حجمه المتوقع؟
 
الخطوة 3 — خطة النسخ الاحتياطي:
| النوع | التكرار | الأداة | مكان التخزين | مدة الاحتفاظ | RPO |
|---|---|---|---|---|---|
| Full Backup | يومياً | Supabase Auto Backup | Supabase Storage | 30 يوم | 24 ساعة |
| Transaction Log | مستمر | PITR | Supabase | 7 أيام | 5 دقائق |
| Snapshot يدوي | قبل Migration | pg_dump | Supabase Storage + External | 90 يوم | - |
| Export البيانات | أسبوعياً | Custom Script | AWS S3 / Supabase | 1 سنة | - |
 
الخطوة 4 — اختبار Disaster Recovery:
صف إجراء اختبار DR خطوة بخطوة:
1. إنشاء بيئة اختبار مستقلة (منفصلة عن الإنتاج)
2. استعادة أحدث نسخة احتياطية
3. قياس وقت الاستعادة الفعلي
4. التحقق من سلامة البيانات (Row Count + Sample Verification)
5. التحقق من عمل الـ RLS Policies في البيئة المستعادة
6. توثيق RTO الفعلي (مقارنةً بالمستهدف ≤ 4 ساعات)
 
الخطوة 5 — Database Maintenance Plan:
```sql
-- يُشغَّل يومياً في 2 AM (ساعة انتفاع منخفض بتوقيت مصر)
VACUUM ANALYZE cases;
VACUUM ANALYZE sessions;
VACUUM ANALYZE documents;
VACUUM ANALYZE invoices;
 
-- يُشغَّل أسبوعياً
REINDEX CONCURRENTLY idx_cases_fulltext;
 
-- مراقبة Table Bloat
SELECT schemaname, tablename,
       pg_size_pretty(pg_total_relation_size(quote_ident(tablename))) as size,
       n_dead_tup, n_live_tup,
       round(100*n_dead_tup::numeric/GREATEST(n_live_tup+n_dead_tup,1), 2) as dead_ratio
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC;
```
