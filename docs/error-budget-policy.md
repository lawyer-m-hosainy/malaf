## 1. الغرض من هذه السياسة

تُحدد هذه السياسة القواعد الرسمية لإدارة Error Budget في منصة مَلَف. الهدف هو
تحقيق توازن واضح بين **سرعة التطوير** و**موثوقية الخدمة**، مع ضمان الامتثال
للمتطلبات القانونية المصرية (ETA / PDPL).

---

## 2. تعريفات SLI/SLO

| SLO | المؤشر (SLI) | الهدف | Error Budget شهرياً |
|-----|-------------|-------|---------------------|
| **Availability** | نسبة HTTP 2xx+3xx من الإجمالي | 99.5% | 0.5% = ~216 دقيقة |
| **Latency P95** | نسبة الطلبات ≤ 500ms | 95% | 5% من الطلبات |
| **Latency P99** | نسبة الطلبات ≤ 2000ms | 99% | 1% من الطلبات |
| **ETA Integration** | نسبة الفواتير المُرسلة بنجاح (3 retries) | 99% | 1% فشل مقبول |

> ⚖️ **ملاحظة قانونية:** ETA Integration SLO مطلوب قانونياً بموجب نظام الفاتورة الإلكترونية المصرية.
> أي تجاوز يستوجب تصعيداً فورياً.

---

## 3. مستويات الاستجابة

### 🟢 المستوى 1: سليم (0–50% من Budget مُستهلك)

| البند | الإجراء |
|-------|---------|
| التطوير | يسير بشكل طبيعي |
| النشر | مسموح بدون قيود |
| الميزات الجديدة | يمكن إضافتها |
| المراجعة | أسبوعية للـ Metrics |
| الفريق | لا يلزم تدخل خاص |

### 🟡 المستوى 2: تحذير (50–90% من Budget مُستهلك)

| البند | الإجراء |
|-------|---------|
| التطوير | تقليل وتيرة الإصدارات |
| النشر | مرة واحدة يومياً فقط |
| الميزات الجديدة | أولوية لـ Bug Fixes |
| المراجعة | يومية للـ Error Rate |
| الفريق | تحليل سبب الاستهلاك المرتفع |
| التنبيهات | إشعار عبر البريد الإلكتروني |

### 🔴 المستوى 3: حرج (90–100% من Budget مُستهلك)

| البند | الإجراء |
|-------|---------|
| التطوير | ❌ إيقاف نشر أي ميزة جديدة |
| النشر | فقط إصلاحات الموثوقية |
| الميزات الجديدة | ❌ ممنوعة تماماً |
| المراجعة | كل 4 ساعات |
| الفريق | كل الطاقة موجهة للموثوقية |
| التنبيهات | بريد + واتساب + Discord |
| Post-Mortem | إلزامي لكل حادثة |

### 🚨 المستوى 4: مُستنفد (> 100% — Budget نفد)

| البند | الإجراء |
|-------|---------|
| التطوير | 🚨 Freeze كامل فوري |
| النشر | 🚫 محظور تماماً |
| الميزات الجديدة | 🚫 محظورة |
| المراجعة | جلسة Emergency Review في 4 ساعات |
| الفريق | خطة استعادة مكتوبة مطلوبة |
| التنبيهات | تصعيد فوري لجميع القنوات |
| الاستئناف | لا يُستأنف التطوير حتى كتابة Post-Mortem + خطة منع تكرار |

## 4. قواعد خاصة بـ SLOs القانونية

تنطبق قواعد أكثر صرامة على **ETA Integration SLO**:

- عتبة التحذير: **30%** (بدلاً من 50%)
- عتبة الحرج: **70%** (بدلاً من 90%)
- أي حالة "حرج" تمنع النشر فوراً
- يجب إخطار المسؤول القانوني عند تجاوز 50%

---

## 5. Burn Rate Alerts

| نوع التنبيه | المعدل | المعنى | القنوات |
|------------|--------|--------|---------|
| **Fast Burn** | ≥ 14.4x | Budget شهر سينفد في ساعة | واتساب + بريد + Discord |
| **Medium Burn** | ≥ 6.0x | Budget شهر سينفد في 5 أيام | بريد + Discord |
| **Slow Burn** | ≥ 2.0x | Budget شهر سينفد في أسبوعين | بريد |
| **Drift** | ≥ 1.0x | Budget ينفد تدريجياً | Dashboard فقط |

## 6. بوابة النشر (Deployment Gate)

يتم فحص SLOs تلقائياً قبل كل عملية نشر:

```
┌─────────────────┐     ┌──────────────┐     ┌──────────────┐
│  Developer Push  │────▶│  CI Pipeline │────▶│  SLO Check   │
└─────────────────┘     └──────────────┘     └──────┬───────┘
                                                     │
                                          ┌──────────┴──────────┐
                                          ▼                     ▼
                                   ┌────────────┐       ┌─────────────┐
                                   │ ✅ Deploy  │       │ ❌ Blocked  │
                                   └────────────┘       └─────────────┘
```

- إذا كان أي SLO في حالة `exhausted` → **النشر محظور**
- إذا كان SLO قانوني في حالة `critical` → **النشر محظور**
- يتم تسجيل كل قرار في جدول `deployment_gate_log`

## 7. المراجعة الشهرية

في أول يوم عمل من كل شهر:

1. **مراجعة تقرير SLO الشهري** — يتضمن كل SLO وحالته
2. **تحليل الحوادث** — كل حادثة مع سببها وتأثيرها
3. **قرار الشهر القادم:**
   - [ ] استئناف التطوير الطبيعي
   - [ ] تقليل وتيرة الـ Deployments
   - [ ] Freeze كامل والتركيز على الموثوقية
4. **تحديث SLO targets** إذا لزم الأمر
