# 🚀 خريطة طريق منصة مَلَف: مجموعة أوامر التشغيل (Super Prompts)

هذا الملف يحتوي على "سوبر برومبت" (أوامر مفصلة وشاملة) لكل ميزة من الميزات الناقصة في منصة مَلَف.
**طريقة الاستخدام:** انسخ البرومبت بالكامل وألصقه في محادثة جديدة معي أو مع أي نموذج AI آخر للبدء في تنفيذ الميزة باحترافية وبدون أي ارتباك.

## أولاً: النواقص القانونية (Legal Features)

### 1. حاسبة المواعيد القانونية (Legal Deadlines Calculator) 🔴 [حرج]

> **البرومبت:**
> "مرحباً! أعمل على منصة SaaS لإدارة مكاتب المحاماة في مصر (مَلَف). أريد بناء وحدة 'حاسبة المواعيد القانونية' (Legal Deadlines Calculator).
> **المطلوب:**
>
> 1. إنشاء `store/useDeadlinesStore.ts` لإدارة الحالة.
> 2. بناء مكون واجهة مستخدم (UI) يسمح للمحامي باختيار 'نوع الإجراء' (مثال: استئناف حكم مدني، طعن بالنقض، معارضة في جنحة).
> 3. إدخال 'تاريخ صدور الحكم' أو 'تاريخ الإعلان'.
> 4. برمجة خوارزمية (Utility function) تحسب تاريخ السقوط بناءً على قانون المرافعات والإجراءات الجنائية المصري (مثال: الاستئناف المدني 40 يوماً، النقض 60 يوماً، المعارضة 10 أيام).
> 5. مراعاة العطلات الرسمية (إذا وافق آخر يوم عطلة يمتد لأول يوم عمل).
> 6. إضافة زر 'حفظ في الأجندة' ليتم إضافة الموعد تلقائياً إلى `useCasesStore` أو المهام مع تنبيه قبل الموعد بـ 3 أيام.
>    يرجى كتابة الكود كاملاً لـ TypeScript (Types) و Zustand Store والـ React Component."

## ثانياً: النواقص التقنية (Technical Features)

### 6. عزل بيانات المكاتب (Row Level Security - RLS) 🔴 [حرج جداً]

> **البرومبت:**
> "أنا أستخدم Supabase كقاعدة بيانات لمنصة SaaS متعددة المستأجرين (Multi-tenant). كل الجداول تحتوي على عمود `organization_id`. أريد كتابة أوامر SQL لتفعيل الـ RLS وحماية البيانات.
> **المطلوب:**
>
> 1. سكربت SQL يفعّل الـ RLS على جداول: `profiles`, `cases`, `clients`, `invoices`, `enforcement_cases`.
> 2. كتابة سياسات (Policies) بحيث لا يمكن لأي مستخدم (Authenticated) القراءة، أو الإضافة، أو التحديث، أو الحذف إلا إذا كان الـ `organization_id` في الصف يطابق الـ `organization_id` الخاص بملف المستخدم (Profile) الحالي.
> 3. شرح كيفية الحصول على الـ `organization_id` للمستخدم الحالي داخل الـ Policy بشكل آمن لتجنب الثغرات."

### 7. إشعارات الموبايل (Push Notifications) 🔴 [حرج]

> **البرومبت:**
> "تطبيقنا هو PWA، وأريد تفعيل الـ Push Notifications لتنبيه المحامين بالجلسات قبل يوم من موعدها.
> **المطلوب:**
>
> 1. إعداد Service Worker في Vite (`vite-plugin-pwa`) للتعامل مع الـ Push events.
> 2. كود لتسجيل المستخدم للحصول على الإشعارات (طلب الإذن `Notification.requestPermission`) وحفظ الـ Subscription Object في Supabase.
> 3. إعداد Supabase Edge Function تُشغَّل دورياً (عبر pg_cron أو أداة خارجية) للبحث عن الجلسات التي ستعقد غداً وإرسال Web Push Notification للأجهزة المسجلة باستخدام بروتوكول VAPID.
>    يرجى توفير الخطوات وملفات الكود المطلوبة بالترتيب."

### 8. الوضع غير المتصل (Offline Mode) 🟡 [مهم]

> **البرومبت:**
> "المحامون يستخدمون المنصة في المحاكم حيث يكون الإنترنت ضعيفاً أو معدوماً. أريد تحسين الـ PWA ليدعم وضع 'Offline' بشكل أفضل.
> **المطلوب:**
>
> 1. تكوين `vite-plugin-pwa` لتخزين واجهة التطبيق كاملة (App Shell).
> 2. استخدام `indexedDB` (أو مكتبة مثل `localforage` أو Zustand Persist) لتخزين بيانات 'قضاياي' و 'جلسات اليوم' محلياً بحيث يمكن عرضها حتى بدون إنترنت.
> 3. تصميم شريط تنبيه (Banner) صغير في التطبيق يظهر عندما ينقطع الإنترنت يخبر المستخدم 'أنت الآن في وضع عدم الاتصال، يمكنك استعراض الجلسات المحفوظة'.
>    يرجى كتابة الـ Config وكود التنبيه."

### 10. خط أنابيب النشر الآلي (CI/CD Pipeline) 🟡 [مهم]

> **البرومبت:**
> "منصتنا تنمو ولا أريد النشر اليدوي الذي قد يسبب أخطاء. المشروع يستخدم React + Vite و Supabase، ومرفوع على Render/Vercel (والكود في GitHub).
> **المطلوب:**
>
> 1. كتابة ملف `github/workflows/main.yml` لإعداد GitHub Actions.
> 2. الـ Pipeline يجب أن يقوم بالتالي عند الدفع (Push) لفرع `main`:
>    - تنزيل الاعتماديات (`npm install`).
>    - فحص الكود (`npm run lint` & Type check).
>    - بناء المشروع (`npm run build`).
> 3. إذا نجح البناء، أريد إشعارات على ديسكورد/سلاك أو مجرد توضيح كيف سيتم النشر التلقائي للخدمة.
>    يرجى كتابة ملف الـ YAML وشرح المتغيرات المطلوبة."
